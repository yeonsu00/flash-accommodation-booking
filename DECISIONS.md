# DECISIONS

## 1. 재고 정합성 - Redis Lua 스크립트로 원자적 실행

### 상황

분산 환경(서버 2대)에서 00시에 500~1000 TPS가 순간적으로 집중됩니다. 재고(10개)를 정확히 지켜야 하고, 초과 판매와 미달 판매 모두 허용할 수 없습니다.

### 선택지

1. **DB 비관적 락 (SELECT FOR UPDATE)** - DB에서 락을 잡고 재고 차감
2. **Redis Lua 스크립트** - Redis에서 재고 조회 + 차감을 원자적으로 실행

### 왜 그렇게 판단했는지

DB 비관적 락은 락 획득 대기열이 DB 내부에 쌓입니다. 1000 TPS 상황에서는 커넥션 풀(최대 30개)이 빠르게 소진되고, 락 경합으로 응답 지연이 커질 수 있습니다.

Redis는 단일 스레드로 명령을 처리합니다. Lua 스크립트는 서버에서 원자적으로 실행되기 때문에 **다른 요청이 중간에 끼어드는 문제 없이** 재고 조회 후 차감을 수행할 수 있습니다. 또한 Redis 응답 속도가 DB보다 훨씬 빠르기 때문에 높은 TPS에서 유리합니다.

## 2. 공정성 - Redis Sorted Set 기반 대기열

### 상황

"모든 사용자에게 동등한 기회"를 제공하되, 먼저 요청한 사람이 먼저 입장해야 합니다. 시작 시점에 500~1000 TPS가 집중되며, 분산 환경(서버 2대)에서는 요청이 서로 다른 서버에 도달하기 때문에 선착순 기준을 명확하게 정의해야 했습니다.

### 선택지

1. **즉시 재고 경쟁 방식** - 사용자가 바로 재고 차감을 시도
2. **Redis Sorted Set 기반 대기열** - 서버 수신 순서대로 대기열을 구성하고 순차적으로 입장 허용

### 왜 그렇게 판단했는지

즉시 재고 경쟁 방식은 구현이 단순하고 처리 속도가 빠르지만, 네트워크 지연 시간이나 서버 도달 속도에 따라 결과가 달라질 수 있습니다. 

Redis Sorted Set은 score를 기준으로 자동 정렬되는 자료구조입니다. 대기열 등록 시 **서버가 요청을 수신한 시각**을 score로 저장하여 시스템 차원에서 공식적인 순서를 기록할 수 있습니다.

이후 Scheduler가 대기열의 앞 순번 사용자를 admitted 상태로 승격시키고, admitted 상태인 사용자만 Checkout 단계로 진입할 수 있도록 설계하였습니다.

## 3. 고가용성 - 대기열 활용

### 상황

인프라 증설이 제한된 상황에서 예약 API가 순간 부하를 직접 처리하면 DB 커넥션 고갈, 결제 시스템 과부하가 발생할 수 있습니다.

### 선택지

1. **Rate Limiting** - Bucket4j 등으로 요청 자체를 거절
2. **대기열(Queue) 기반 흐름 제어** - 요청을 대기열에 담고 일정 속도로 처리

### 왜 그렇게 판단했는지

Rate Limiting은 초과 요청을 단순히 거절하기 때문에 사용자 경험이 나쁘고, 거절된 사용자가 재시도를 반복하면 오히려 부하가 증가합니다.

대기열 방식은 00시에 몰린 요청을 Redis Sorted Set에 모두 등록한 뒤, `QueueAdmissionScheduler`가 100ms마다 5명씩 예약 단계로 넘깁니다. 그래서 예약 API는 안정적인 트래픽만 받게 됩니다. 사용자는 거절당하지 않고 "대기 중" 상태로 기다리며, 순서가 되면 자동으로 입장됩니다.

스케줄러는 Redisson 분산 락(`tryLock`)으로 2대 서버에서 중복 실행을 방지합니다.

## 4. 결제 확장성 - Strategy 패턴 + Factory 자동 등록

### 상황

신용카드, Y페이, Y포인트를 지원하고 향후 새로운 결제 수단이 추가될 수 있습니다. Booking API의 비즈니스 로직 수정을 최소화하는 구조를 도입해야 합니다.

### 선택지

1. **if-else / switch 분기** - 결제 수단마다 조건 분기
2. **Strategy 패턴 + Factory** - `PaymentProcessor` 인터페이스 구현체를 Factory에서 조회

### 왜 그렇게 판단했는지

if-else 방식은 새 결제 수단 추가 시 `PaymentService` 내부를 직접 수정해야 합니다. 기존 로직을 반복적으로 변경하는 과정에서 조건 누락이나 잘못된 수정과 같은 휴먼 에러가 발생할 수 있습니다.

`PaymentProcessorFactory`는 Spring이 주입하는 `List<PaymentProcessor>`를 `PaymentMethodType → Processor` 맵으로 변환합니다. 새 결제 수단은 `PaymentProcessor`를 구현하는 클래스를 추가하고 `@Component`를 붙이기만 하면 Factory에 자동 등록됩니다. `BookingFacade`와 `PaymentService`는 전혀 수정하지 않습니다.

복합 결제는 결제 수단 목록을 순회하며 각 Processor를 호출하는 방식으로 처리합니다.

## 5. 멱등성 - Redis SET NX + DB UNIQUE 이중 방어

### 상황

주문서에서 짧은 간격으로 결제 버튼이 연속 클릭되거나 네트워크 재전송으로 중복 요청이 들어올 수 있습니다. 동일 요청이 두 번 처리되면 결제가 중복으로 청구됩니다.

### 선택지

1. **DB UNIQUE 제약만 사용** - `payment.idempotency_key`에 UNIQUE 제약
2. **Redis SET NX만 사용** - Redis 장애 시 무방비
3. **Redis SET NX를 1차, DB UNIQUE를 2차 방어로 사용**

### 왜 그렇게 판단했는지

DB UNIQUE만 사용하면 멱등성 확인을 위해 매 요청마다 DB를 거쳐야 합니다. 결제 요청이 집중되는 00시에는 DB 부하가 증가합니다.

Redis SET NX만 사용하면 Redis 장애 시 중복 요청이 모두 통과됩니다.

Redis SET NX를 1차, DB UNIQUE를 2차 방어로 사용했습니다. Redis 정상 시에는 SET NX가 빠르게 1차 차단하고, Redis 장애 시에는 fallback이 `true`를 반환해 요청을 허용하되 DB의 UNIQUE 제약이 2차로 막습니다. DB 저장 시 `DataIntegrityViolationException`이 발생하면 `BusinessException(DUPLICATE_PAYMENT)`으로 예외처리합니다.

## 6. Redis 장애 대응

### 상황

Redis가 장애 상태일 때 모든 기능에서 예외를 던지면 재고가 남아 있어도 예약이 불가능해집니다.

### 기능별 장애 대응 방식

기능별로 폴백 전략을 다르게 적용합니다.

| 기능 | Redis 장애 시 폴백 | 이유 |
|------|-------------------|------|
| 재고 선점 | DB FOR UPDATE | DB로 정합성 보장 가능 |
| 대기열 | 503 반환 | Sorted Set 없이 공정한 순서 보장 불가 |
| checkoutToken 조회 | DB checkout_token 조회 | 재고 선점 시 이중 저장했으므로 가능 |
| 멱등성 | 요청 허용 → DB UNIQUE 2차 방어 | 최소한의 중복 방지 유지 |

### 왜 그렇게 판단했는지

재고 선점은 예약 시스템의 핵심 기능이고, 무엇보다 정합성이 중요합니다. Redis를 사용할 수 없는 상황에서는 성능 저하를 감수하더라도 DB 비관적 락을 통해 초과 판매를 방지하는 방법을 선택했습니다.

대기열은 공정성을 보장하기 위한 기능입니다. Redis Sorted Set 없이 RDB 기반으로 대기열을 임시 구현하는 방법도 고려했지만, 500~1000 TPS 환경에서 DB 락 경합과 커넥션 풀 고갈 가능성이 있을 수 있다고 생각했습니다. 그래서 Redis 장애 시에는 대기열 기능을 비활성화하고 503 응답을 반환하여, 불완전한 방식으로 서비스를 제공하는 것보다 명시적으로 실패시키는 방향을 선택했습니다.

checkoutToken과 멱등성은 별도의 DB 저장소를 활용하여 최소한의 기능을 유지하도록 설계했습니다. 이를 통해 Redis 장애 상황에서도 이미 주문서를 발급받은 사용자의 예약 진행을 지원하고, 중복 결제 위험을 최소화할 수 있도록 하였습니다.

## 7. PG API 장애 대응 - Retry와 CircuitBreaker 조합

### 상황

PG사 연동은 일시적 네트워크 오류(재시도로 해결 가능)와 PG사 시스템 장애(재시도해도 의미 없음)를 구분해야 합니다. 무한 재시도는 오히려 장애를 악화시킵니다.

### 선택지

1. **Retry만 적용** - 장애 지속 시에도 계속 재시도하여 PG 부하 가중
2. **CircuitBreaker만 적용** - 일시적 오류를 재시도 없이 즉시 실패 처리
3. **Retry + CircuitBreaker 모두 적용** - 재시도로 일시적 오류 흡수, 지속 장애 시 서킷 OPEN

### 왜 그렇게 판단했는지

Retry만 적용하면 PG가 장기 장애일 때도 계속 재시도합니다. 타임아웃이 누적되어 스레드가 묶이고 PG에 불필요한 부하가 가중되어 장애를 오히려 악화시킵니다.

CircuitBreaker만 적용하면 재시도 한 번으로 해결될 수 있는 일시적인 네트워크 오류도 즉시 실패 처리할 수 있습니다.

두 가지를 조합하면 역할이 분리됩니다. Retry가 일시적 오류를 흡수하고, 반복 실패로 실패율이 임계치를 넘으면 CircuitBreaker가 열려 이후 요청은 PG 호출 없이 즉시 거절합니다. PG 장애가 예약 시스템 전체로 전파되는 것을 막고, PG 복구 후에는 HALF_OPEN을 거쳐 자동으로 CLOSED 상태로 돌아옵니다.

**테스트 가능성을 위한 구조 분리**: `@CircuitBreaker` + `@Retry`가 붙은 빈에 `@MockitoSpyBean`을 걸면 Spring AOP 프록시가 교체되어 Resilience4j가 동작하지 않습니다. 이를 해결하기 위해 랜덤 결과 시뮬레이션 로직(`PgPaymentSimulator`)과 Resilience4j 적용 레이어(`PgPaymentClient`)를 분리했습니다. 테스트에서는 `PgPaymentSimulator`만 Spy하여 AOP를 온전히 유지합니다.

## 8. 결제 실패 보상 - 이벤트 기반 보상 트랜잭션

### 상황

복합 결제(예: 신용카드 + Y포인트)에서 카드 승인 후 포인트 차감이 실패하면, 이미 승인된 카드 결제를 취소해야 합니다. 취소 처리를 트랜잭션 내에서 직접 수행하면 구조가 복잡해집니다.

### 선택지

1. **동기 보상** - catch 블록에서 직접 취소 로직 실행
2. **이벤트 기반 보상** - `PaymentCompensationEvent`를 발행하고 `AFTER_ROLLBACK` 리스너에서 처리

### 왜 그렇게 판단했는지

catch 블록에서 직접 취소 로직을 실행하면 비즈니스 로직과 보상 로직이 섞여 가독성이 낮아집니다.

`@TransactionalEventListener(phase = AFTER_ROLLBACK)`는 트랜잭션이 롤백된 직후에만 실행됩니다. 보상이 필요한 경우(예외 발생)에만 동작하며, DB 롤백이 완료된 이후에 PG 취소가 실행되므로 순서가 명확합니다. `BookingCompletedEvent(AFTER_COMMIT)`도 동일한 방식으로 커밋 후에만 실행되어 checkoutToken 삭제와 멱등성 결과 저장이 트랜잭션 외부에서 처리됩니다.

## 9. Redisson 도입

### 상황

`QueueAdmissionScheduler`는 2대 서버에서 동시에 실행됩니다. 분산 락 없이는 같은 대기열 토큰이 두 번 입장 처리될 수 있습니다.

### 선택지

1. **Spring Data Redis (Lettuce) + SET NX로 직접 구현** - TTL 관리, 락 해제 보장 등 직접 구현 필요
2. **Redisson** - 분산 락 구현체 제공, Watchdog으로 자동 TTL 연장

### 왜 그렇게 판단했는지

SET NX 기반 직접 구현은 락 보유 중 서버가 죽으면 TTL 만료까지 락이 유지됩니다. 반대로 TTL을 짧게 설정하면 처리 중에 락이 풀릴 수 있습니다. 락을 보유한 스레드가 아닌 다른 스레드가 해제하는 문제도 직접 방어해야 합니다.

Redisson의 `RLock`은 Watchdog이 락 보유 동안 자동으로 TTL을 연장합니다. `tryLock()`은 락 획득 실패 시 즉시 `false`를 반환하므로 100ms 간격의 스케줄러에서 블로킹 없이 사용할 수 있습니다. `unlock()`은 락을 보유한 스레드만 해제할 수 있도록 내부적으로 Lua 스크립트로 처리합니다.

## 10. checkoutToken 이중 저장 (Redis + DB)

### 상황

재고 선점 후 발급된 checkoutToken은 Redis에 5분 TTL로 저장됩니다. 예약 API 호출 시 이 토큰으로 `userId`와 `productId`를 조회합니다. Redis 장애 시 토큰을 찾지 못하면, 재고는 선점했지만 예약을 완료할 수 없는 상황이 됩니다.

### 선택지

1. **Redis만 저장** - 장애 시 토큰 유실, 예약 불가
2. **DB만 저장** - 매 요청마다 DB 조회 필요
3. **Redis + DB 이중 저장** - Redis 우선 조회, 장애 시 DB 폴백

### 왜 그렇게 판단했는지

정상 경로는 Redis에서 O(1)으로 조회합니다. Redis 장애 시에는 `checkout_token` 테이블에서 조회합니다. DB 조회 시 `expiredAt` 컬럼으로 Redis TTL과 동일한 만료 검증을 적용하여 만료된 토큰을 사용할 수 없도록 합니다.

이중 저장의 비용은 재고 선점 시 DB INSERT 1회 추가입니다. 이는 선점 당 1회 발생하는 비용으로, 정상 예약 흐름에서 발생하는 DB 부하에 비해 미미합니다. Redis 장애라는 드문 상황에서도 이미 재고를 선점한 사용자가 예약을 완료할 수 있다는 점에서 비용 대비 효과가 충분하다고 생각합니다.
